**Descripción:**
Como docente
Quiero que al publicar mi programación didáctica el sistema genere automáticamente el PDF oficial de forma asíncrona en segundo plano
Para recibir el documento listo sin que el proceso de publicación se bloquee esperando la generación del PDF.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Encolar Job al publicar):**
  **Dado que** un documento pasa a estado Publicado,
  **Cuando** la publicación se confirma,
  **Entonces** se encola automáticamente un `GeneratePdfJob` en RabbitMQ con el `document_version_id`; la respuesta HTTP al usuario se devuelve en menos de 50 ms sin esperar la generación.

- [ ] **Escenario 2 (Validación de permisos en el Worker):**
  **Dado que** el Worker consume el `GeneratePdfJob` de la cola,
  **Cuando** procesa el Job,
  **Entonces** valida que el `document_version_id` del Job corresponde a una versión realmente publicada antes de invocar Puppeteer; si no existe, descarta el Job y registra el error.

- [ ] **Escenario 3 (Renderizado PDF con Puppeteer/Chromium):**
  **Dado que** el Job es válido y autorizado,
  **Cuando** el Worker invoca el motor PDF,
  **Entonces** Puppeteer renderiza el HTML generado por el backend (con metadatos F-08.3) a PDF usando Chromium headless en contenedor aislado.

- [ ] **Escenario 4 (Reintentos con backoff exponencial):**
  **Dado que** la generación del PDF falla por un error temporal (Chromium no disponible, timeout),
  **Cuando** el Job falla,
  **Entonces** se reintenta automáticamente hasta 3 veces con backoff exponencial (30 s, 90 s, 270 s); tras el tercer fallo, el Job se marca como fallido y se registra el error.

- [ ] **Escenario 5 (Límite de instancias Chromium):**
  **Dado que** múltiples Jobs de PDF se procesan simultáneamente,
  **Cuando** ya hay 5 instancias de Chromium activas,
  **Entonces** los Jobs adicionales esperan en la cola hasta que se libere una instancia; no se generan más de 5 procesos Chromium en paralelo.

- [ ] **Requisito Técnico (Latencia de publicación):**
  La respuesta HTTP al endpoint de publicación es inferior a 50 ms; la generación del PDF ocurre en segundo plano y no bloquea al usuario.

**Notas:**
- Chromium se ejecuta en un contenedor Docker aislado sin acceso a la red interna para mitigar vulnerabilidades de sandbox (STRIDE E-PDF-01).
- `PdfJobService::dispatch(string $documentVersionId)` encola el Job; el `GeneratePdfJob` invoca `PdfGenerationService::generate()`.
- El HTML para el PDF se genera en el backend (no desde el frontend) para prevenir inyección de datos falsos (STRIDE T-PDF-01).
- La actualización del estado del PDF (pending/processing/ready/failed) se almacena en `document_pdf_status` para consulta desde la API (F-08.2).
