**Descripción:**
Como docente o revisor
Quiero ver en mi pantalla de inicio una vista unificada de mis acciones pendientes más importantes categorizadas por tipo
Para saber exactamente qué tengo que hacer al entrar a Maya PD sin tener que navegar por múltiples secciones.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Endpoint BFF devuelve datos agregados):**
  **Dado que** el usuario autenticado accede al dashboard,
  **Cuando** la pantalla carga,
  **Entonces** una única llamada a `GET /api/dashboard` devuelve: borradores del usuario, documentos pendientes de su revisión como validador y documentos próximos a vencer, sin necesidad de múltiples llamadas.

- [ ] **Escenario 2 (Tarjetas de acción con badges semánticos):**
  **Dado que** el endpoint BFF ha devuelto los datos,
  **Cuando** el frontend renderiza el dashboard,
  **Entonces** cada ítem se muestra como una tarjeta con: badge semántico ("Urgente", "Revisión pendiente", "Continuar borrador"), título del documento, módulo y fecha límite si aplica.

- [ ] **Escenario 3 (Ordenación por urgencia):**
  **Dado que** el dashboard tiene varios ítems de distintos tipos,
  **Cuando** se renderizan las tarjetas,
  **Entonces** los ítems marcados como "Urgente" (próximos a vencer o vencidos) aparecen primero, seguidos por los pendientes de revisión y luego los borradores en progreso.

- [ ] **Escenario 4 (Clic en tarjeta redirige al documento):**
  **Dado que** el usuario visualiza una tarjeta de acción,
  **Cuando** hace clic en ella,
  **Entonces** es redirigido directamente al editor del documento (si es borrador) o a la vista de revisión (si está en revisión), sin pasos intermedios.

- [ ] **Escenario 5 (Refresco automático):**
  **Dado que** el usuario tiene el dashboard abierto,
  **Cuando** transcurren 5 minutos,
  **Entonces** los datos del dashboard se actualizan automáticamente sin recargar la página completa.

- [ ] **Requisito Técnico (Respuesta del endpoint BFF):**
  El endpoint `GET /api/dashboard` debe responder en menos de 300 ms en el percentil 95 con los índices correctos sobre `(creator_id, status)`, `(reviewer_id, status)` y `(deadline_date)`.

**Notas:**
- El dashboard es una query directa a BD; no usa notificaciones push ni WebSocket (Restricción 8).
- `DashboardService::getSummary(string $userId)` agrega las tres consultas y devuelve un `DashboardSummaryDTO { pending_drafts[], pending_reviews[], urgent_items[] }`.
- El `DashboardController` recibe el DTO del Service y lo pasa al `DashboardResource` para la serialización JSON.
- Las tarjetas "Urgente" se determinan por `deadline_date <= NOW() + 3 días`.
