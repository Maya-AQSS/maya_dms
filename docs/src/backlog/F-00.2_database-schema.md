**Descripción:**
Como arquitecto de datos
Quiero definir el esquema base de PostgreSQL 16 con todas las migraciones iniciales (plantillas, documentos, bloques JSONB, snapshots, grupos, audit_log)
Para que los modelos de negocio tengan una base sólida e indexada desde el primer sprint.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Migraciones completas):**
  **Dado que** el entorno de base de datos está disponible,
  **Cuando** se ejecuta el comando de migración,
  **Entonces** se crean las tablas: `templates`, `template_versions`, `documents`, `document_versions`, `blocks`, `groups`, `group_user`, `template_assignments`, `document_collaborators`, `audit_log` y `comments`, todas con UUID como clave primaria.

- [ ] **Escenario 2 (Índices de rendimiento):**
  **Dado que** las tablas están creadas,
  **Cuando** se inspeccionan los índices de la base de datos,
  **Entonces** existen índices B-Tree en las columnas de filtrado frecuente: `study_id`, `year`, `status` y `creator_id` en las tablas correspondientes.

- [ ] **Escenario 3 (Restricción de audit_log):**
  **Dado que** el usuario de aplicación está configurado en la base de datos,
  **Cuando** intenta ejecutar una operación UPDATE o DELETE sobre la tabla `audit_log`,
  **Entonces** la operación es rechazada por el sistema de permisos de base de datos.

- [ ] **Escenario 4 (Almacenamiento JSONB de bloques):**
  **Dado que** el esquema está definido,
  **Cuando** se revisa la estructura de `document_versions`,
  **Entonces** los bloques de contenido se almacenan como campo JSONB dentro de `document_versions`, no como filas individuales en una tabla separada.

- [ ] **Requisito Técnico (Ejecución limpia):**
  El comando `php artisan migrate` debe ejecutarse sin errores en un servidor PostgreSQL 16 limpio sin datos previos.
