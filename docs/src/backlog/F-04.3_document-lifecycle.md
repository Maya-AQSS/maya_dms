**Descripción:**
Como autor de una programación didáctica
Quiero gestionar el ciclo de vida de mi documento (Borrador → En Revisión → Publicado) con el flujo de validadores configurado en la plantilla origen
Para garantizar la trazabilidad y la aprobación formal requerida por ISO 9001.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Envío a revisión con validadores):**
  **Dado que** un documento en Borrador tiene todos los bloques obligatorios rellenos y la plantilla tiene N validadores,
  **Cuando** el autor principal pulsa "Enviar a revisión",
  **Entonces** el documento pasa a estado `En Revisión`, queda bloqueado para edición del autor y se emite evento a RabbitMQ para notificar al validador correspondiente.

- [ ] **Escenario 2 (Publicación directa sin validadores):**
  **Dado que** la plantilla del documento tiene 0 validadores configurados,
  **Cuando** el autor pulsa "Publicar",
  **Entonces** el documento pasa directamente de Borrador a Publicado; se crea el snapshot (F-04.4) y se registra la transición en `audit_log`.

- [ ] **Escenario 3 (Estado Validado intermedio — publicación manual):**
  **Dado que** la plantilla tiene configurada publicación manual y el último validador aprueba,
  **Cuando** la última aprobación se registra,
  **Entonces** el documento pasa a estado `Validado`; solo el autor principal puede pulsar "Publicar" para moverlo a `Publicado`.

- [ ] **Escenario 4 (Documento bloqueado en revisión):**
  **Dado que** un documento está en estado `En Revisión`,
  **Cuando** el autor intenta modificar el contenido de un bloque,
  **Entonces** el sistema devuelve HTTP 403 y muestra un mensaje indicando que el documento está en revisión y no puede editarse.

- [ ] **Escenario 5 (SoD en el envío a revisión):**
  **Dado que** el autor de un documento intenta asignarse a sí mismo como validador,
  **Cuando** se configura o se intenta validar,
  **Entonces** el sistema devuelve HTTP 403 con mensaje de violación de Segregación de Funciones.

- [ ] **Escenario 6 (Snapshot al publicar):**
  **Dado que** un documento pasa a estado Publicado,
  **Cuando** la publicación se confirma,
  **Entonces** se crea de forma asíncrona un snapshot inmutable del documento (F-04.4) y se encola el Job de generación PDF (F-08.1).

- [ ] **Requisito Técnico (Transición de estado):**
  La transición de estado se registra en `audit_log` con timestamp del servidor en menos de 100 ms; la emisión del evento a RabbitMQ en menos de 50 ms adicionales.

**Notas:**
- La máquina de estados la gestiona `DocumentService::transition(string $documentId, string $newState, string $actorId)`.
- El número y tipo de validadores lo hereda el documento de la `template_version` con la que fue creado.
- `SubmitForReviewDTO { document_id, submitted_by }` es el DTO de entrada al `DocumentService`.
- El snapshot y el Job de PDF se despachan como eventos de dominio tras la transición, no de forma síncrona.
