**Descripción:**
Como coordinador de calidad
Quiero gestionar el ciclo de vida completo de una plantilla (Borrador → En Revisión → Publicado) con entre 0 y N validadores configurables y modo síncrono o asíncrono
Para garantizar que ninguna plantilla normativa llega a los docentes sin la aprobación necesaria.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Publicación directa sin validadores):**
  **Dado que** una plantilla tiene 0 validadores configurados,
  **Cuando** el creador pulsa "Publicar",
  **Entonces** la plantilla pasa directamente de Borrador a Publicado, se crea el snapshot (F-03.4) y se registra la transición en `audit_log`.

- [ ] **Escenario 2 (Envío a revisión con validadores):**
  **Dado que** una plantilla tiene N validadores configurados,
  **Cuando** el creador pulsa "Enviar a revisión",
  **Entonces** la plantilla pasa a estado `En Revisión`, queda bloqueada para edición del creador y se emite evento a RabbitMQ para notificar al primer validador (síncrono) o a todos (asíncrono).

- [ ] **Escenario 3 (Validación síncrona — orden estricto):**
  **Dado que** la plantilla tiene validación síncrona configurada y el validador 1 aprueba,
  **Cuando** la aprobación se registra,
  **Entonces** se emite evento a RabbitMQ para notificar al validador 2; el validador 2 no recibía el documento hasta este momento.

- [ ] **Escenario 4 (Validación asíncrona — paralelo):**
  **Dado que** la plantilla tiene validación asíncrona configurada,
  **Cuando** se envía a revisión,
  **Entonces** todos los validadores reciben simultáneamente la notificación y pueden revisar en cualquier orden.

- [ ] **Escenario 5 (Rechazo reinicia validaciones):**
  **Dado que** cualquier validador rechaza la plantilla,
  **Cuando** el rechazo se confirma con motivo obligatorio,
  **Entonces** la plantilla vuelve a estado Borrador, todas las validaciones pendientes se cancelan y el creador puede editar y reenviar.

- [ ] **Escenario 6 (SoD — creador no puede validar su propia plantilla):**
  **Dado que** el creador de una plantilla está configurado como validador de la misma,
  **Cuando** intenta aprobar o rechazar,
  **Entonces** el sistema devuelve HTTP 403 y registra el intento en `audit_log`.

- [ ] **Escenario 7 (Publicación manual — estado Validado intermedio):**
  **Dado que** la plantilla tiene configurada publicación manual,
  **Cuando** el último validador aprueba,
  **Entonces** la plantilla pasa a estado `Validado`; solo el creador puede pulsar "Publicar" para moverla a `Publicado`.

- [ ] **Requisito Técnico (Transición de estado):**
  Cada transición de estado se registra en `audit_log` con timestamp del servidor en menos de 100 ms; la emisión del evento a RabbitMQ ocurre en menos de 50 ms adicionales.

**Notas:**
- La máquina de estados la gestiona `TemplateService::transition(string $templateId, string $newState, string $actorId)`.
- El número y tipo de validadores (síncrono/asíncrono) se configuran en la plantilla; se almacenan en la tabla `template_validators` con orden de secuencia.
- Los documentos ya creados con versiones anteriores siguen funcionando aunque la plantilla esté en revisión; se crean sobre la última versión publicada.
- La publicación automática es el comportamiento por defecto (sin estado `Validado` intermedio).
