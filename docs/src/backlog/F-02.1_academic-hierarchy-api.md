**Descripción:**
Como docente
Quiero que la aplicación cargue la jerarquía académica completa (Tipos de Estudio → Estudios → Módulos) al iniciar la sesión
Para poder navegar y filtrar mis programaciones didácticas sin recargar la página.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Endpoint de jerarquía):**
  **Dado que** el usuario ha iniciado sesión correctamente,
  **Cuando** el frontend realiza la petición `GET /api/hierarchy`,
  **Entonces** el servidor devuelve el árbol completo de la jerarquía académica en una sola respuesta JSON anidada, con Tipos de Estudio como nodo raíz, Estudios como hijos y Módulos como hojas.

- [ ] **Escenario 2 (Caché de jerarquía):**
  **Dado que** la jerarquía académica rara vez cambia,
  **Cuando** el endpoint es consultado,
  **Entonces** los datos se sirven desde caché Redis con TTL de 1 hora, evitando consultas repetidas al FDW o al sistema corporativo.

- [ ] **Escenario 3 (Carga única en cliente):**
  **Dado que** el frontend ha descargado la jerarquía al inicio de sesión,
  **Cuando** el usuario navega entre secciones de la aplicación,
  **Entonces** el frontend no vuelve a solicitar la jerarquía al servidor hasta que el usuario recarga la página completa.

- [ ] **Escenario 4 (Selectores en cascada funcionales):**
  **Dado que** la jerarquía está disponible en el cliente,
  **Cuando** el usuario selecciona un Tipo de Estudio en el primer selector,
  **Entonces** el segundo selector se habilita mostrando únicamente los Estudios que pertenecen a ese Tipo; al seleccionar un Estudio, el tercer selector muestra solo sus Módulos.

- [ ] **Requisito Técnico (Rendimiento del endpoint):**
  La respuesta del endpoint `GET /api/hierarchy` debe ser inferior a 300 ms y el payload no debe superar 50 KB para la jerarquía de un centro educativo mediano.
