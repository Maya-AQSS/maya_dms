**Descripción:**
Como equipo de desarrollo
Quiero un pipeline CI/CD en GitHub Actions que ejecute lint, tests automatizados y despliegue a staging y producción
Para garantizar que ningún código roto llega a los entornos compartidos.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Workflow de integración continua):**
  **Dado que** un desarrollador abre un Pull Request,
  **Cuando** el workflow `ci.yml` se ejecuta,
  **Entonces** se ejecutan secuencialmente: lint PHP (PHP CS Fixer), lint JavaScript (ESLint), tests unitarios de Laravel (PHPUnit) y tests unitarios de React (Vitest); el PR no puede mergearse si alguno falla.

- [ ] **Escenario 2 (Despliegue automático a staging):**
  **Dado que** un Pull Request es mergeado a la rama `main`,
  **Cuando** el workflow `deploy-staging.yml` se activa,
  **Entonces** el sistema despliega automáticamente la nueva versión al entorno de staging con datos anónimos.

- [ ] **Escenario 3 (Despliegue a producción por tag):**
  **Dado que** el equipo crea un tag con formato semántico `v*.*.*`,
  **Cuando** el workflow `deploy-prod.yml` se activa,
  **Entonces** el sistema despliega a producción únicamente si todos los tests del CI previo pasaron correctamente.

- [ ] **Escenario 4 (Gestión de secretos):**
  **Dado que** los workflows necesitan credenciales de entorno,
  **Cuando** se revisa el repositorio de código,
  **Entonces** todos los secretos (tokens, claves, passwords) están almacenados en GitHub Secrets y ninguno aparece en los archivos del repositorio.

- [ ] **Escenario 5 (Protección de despliegue):**
  **Dado que** el pipeline CI detecta tests fallidos,
  **Cuando** el workflow de despliegue intenta ejecutarse,
  **Entonces** el despliegue es bloqueado automáticamente hasta que los tests pasen.

- [ ] **Requisito Técnico (Duración del pipeline CI):**
  El pipeline de integración continua completo (lint + tests backend + tests frontend) debe completarse en menos de 10 minutos.

**Notas:**
- En el entorno de staging, la base de datos contiene un volcado sanitizado y anónimo de producción (sin datos personales reales).
- El pipeline nunca despliega si hay tests fallidos, independientemente del tipo de despliegue (staging o producción).
