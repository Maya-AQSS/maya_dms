**Descripción:**
Como sistema backend
Quiero que cada request a la API valide el JWT corporativo y extraiga los claims de sesión del usuario autenticado
Para que todas las operaciones posteriores tengan un contexto de identidad confiable sin depender de la base de datos local en cada petición.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Identificación del usuario):**
  **Dado que** el middleware valida un JWT correctamente,
  **Cuando** la aplicación necesita identificar al usuario,
  **Entonces** el claim `sub` del JWT identifica unívocamente al usuario en Maya PD y está disponible en toda la aplicación durante esa petición.

- [ ] **Escenario 2 (Claims de sesión sin query a BD):**
  **Dado que** el usuario ha sido autenticado,
  **Cuando** se accede al objeto `Auth::user()` de Laravel,
  **Entonces** los claims de sesión (nombre, departamento) están disponibles desde el payload JWT cacheado en Redis, sin ejecutar ninguna query a la base de datos local.

- [ ] **Escenario 3 (Token expirado durante sesión activa):**
  **Dado que** el token JWT de un usuario expira mientras tiene una sesión activa,
  **Cuando** ese usuario realiza la siguiente petición a la API,
  **Entonces** el sistema devuelve HTTP 401 con la cabecera `WWW-Authenticate: Bearer` indicando que se debe renovar el token.

- [ ] **Escenario 4 (Logs de fallo de autenticación):**
  **Dado que** se intenta acceder a la API con un token inválido,
  **Cuando** el sistema registra el evento de seguridad,
  **Entonces** el log incluye la IP de origen, el User-Agent y los primeros 8 caracteres del token (no el token completo) con nivel de severidad WARNING.

- [ ] **Requisito Técnico (Velocidad de resolución de identidad):**
  El tiempo de resolución de la identidad del usuario debe ser inferior a 2 ms en el percentil 95 cuando el caché Redis está activo.

**Notas:**
- El endpoint JWKS del dashboard corporativo se cachea en Redis. Si Redis cae, el sistema valida contra el endpoint remoto como fallback con un timeout máximo de 2 segundos.
- Esta feature formaliza el comportamiento del middleware de F-00.4 en el contexto de la lógica de negocio de Sprint 1.
