**Descripción:**
Como responsable de calidad ISO 9001
Quiero que el sistema aplique políticas de Segregación de Funciones (SoD) en documentos y plantillas
Para garantizar que ningún usuario puede crear y aprobar el mismo artefacto, cumpliendo con los requisitos de auditoría.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (SoD en documentos):**
  **Dado que** un usuario autenticado es el creador de un documento,
  **Cuando** intenta acceder a la acción de revisión o aprobación de ese mismo documento,
  **Entonces** la Policy `DocumentPolicy::review()` retorna una respuesta de acceso denegado (HTTP 403).

- [ ] **Escenario 2 (SoD en plantillas):**
  **Dado que** un usuario autenticado es el creador de una plantilla,
  **Cuando** intenta acceder a la acción de revisión o aprobación de esa misma plantilla,
  **Entonces** la Policy `TemplatePolicy::review()` aplica la misma restricción de segregación y retorna HTTP 403.

- [ ] **Escenario 3 (SoD con delegación):**
  **Dado que** un documento ha sido delegado a un usuario,
  **Cuando** ese usuario delegado como autor intenta revisar o aprobar ese mismo documento,
  **Entonces** la restricción SoD aplica igualmente: el autor delegado no puede auto-aprobarse.

- [ ] **Escenario 4 (Registro de intentos de violación):**
  **Dado que** un usuario intenta violar la restricción SoD,
  **Cuando** la Policy detecta el intento,
  **Entonces** se genera un registro en `audit_log` con nivel WARNING incluyendo el ID del usuario, el ID del artefacto y el timestamp del servidor.

- [ ] **Escenario 5 (Test unitario de verificación):**
  **Dado que** existe un test automatizado de SoD,
  **Cuando** un usuario creador del documento ejecuta la acción de envío a revisión sobre su propio documento,
  **Entonces** el test verifica que la respuesta es HTTP 403 y el test pasa en el pipeline CI.

- [ ] **Requisito Técnico (Velocidad de verificación):**
  La verificación SoD debe ejecutarse en menos de 1 ms ya que es una comparación de IDs sin necesidad de consulta a base de datos.

**MoSCoW:** `Must`
**Sprint:** Sprint 1

**Notas:**
- Esta Policy se aplica independientemente del nivel de acceso o rol del usuario en el sistema.
- En versiones futuras (V2), las Policies se extenderán para incorporar roles adicionales provenientes del JWT corporativo.
