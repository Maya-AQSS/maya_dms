**Descripción:**
Como revisor de una programación didáctica
Quiero rechazar un documento con un motivo obligatorio
Para que el autor entienda qué debe corregir y el sistema reinicie el proceso de validación de forma limpia y completamente trazable.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Rechazo con motivo obligatorio):**
  **Dado que** el revisor ha revisado el documento y decide rechazarlo,
  **Cuando** pulsa "Rechazar" e introduce el motivo,
  **Entonces** el sistema valida que el motivo tenga al menos 20 caracteres; si el campo está vacío o es demasiado corto, no permite confirmar el rechazo.

- [ ] **Escenario 2 (Documento vuelve a Borrador):**
  **Dado que** el rechazo se confirma con motivo válido,
  **Cuando** la acción se procesa,
  **Entonces** el documento vuelve al estado `Borrador`, todas las validaciones pendientes se cancelan y se reinician desde cero; el autor recupera acceso de edición.

- [ ] **Escenario 3 (Motivo visible para el autor):**
  **Dado que** un documento ha sido rechazado,
  **Cuando** el autor abre el documento o su historial,
  **Entonces** el motivo del rechazo está visible con el nombre del revisor y la fecha del rechazo.

- [ ] **Escenario 4 (Registro en auditoría):**
  **Dado que** el rechazo se procesa,
  **Cuando** la transición de estado se completa,
  **Entonces** se registra en `audit_log` el ID del revisor, el ID del documento, el texto del motivo, el estado anterior (`En Revisión`) y el nuevo estado (`Borrador`), con timestamp del servidor.

- [ ] **Escenario 5 (Contador de iteraciones visible):**
  **Dado que** un documento ha sido rechazado y el autor lo reenvía a revisión,
  **Cuando** el revisor visualiza el documento,
  **Entonces** puede ver cuántas iteraciones de revisión ha tenido el documento (ej. "Iteración 2 de revisión").

- [ ] **Escenario 6 (Evento de rechazo emitido a RabbitMQ):**
  **Dado que** el rechazo se confirma,
  **Cuando** la transición se registra,
  **Entonces** se emite un evento `DocumentRejected` a RabbitMQ para que el servicio externo notifique al autor.

- [ ] **Requisito Técnico (Proceso completo de rechazo):**
  La vuelta a Borrador incluyendo cancelación de validaciones y registro en `audit_log` debe completarse en menos de 200 ms.

**MoSCoW:** `Must`
**Sprint:** Sprint 4

**Notas:**
- No hay rechazos parciales (un validador rechaza solo una sección): el rechazo es total y reinicia todas las validaciones. Los rechazos parciales están diferidos a post-MVP (D-08).
- `ReviewService::rejectDocument(RejectDocumentDTO { document_id, reviewer_id, reason })` gestiona la transición de estado, el registro en `audit_log` y la emisión del evento.
- El motivo del rechazo se almacena en la tabla `document_rejections` con `(document_id, reviewer_id, reason, iteration_number, created_at)` para el historial de iteraciones.
