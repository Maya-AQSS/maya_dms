**Descripción:**
Como responsable de calidad ISO 9001
Quiero que todas las acciones relevantes sobre documentos, plantillas, bloques y comentarios queden registradas en una tabla de auditoría relacional separada del contenido JSONB
Para satisfacer los requisitos de trazabilidad completa de la normativa ISO 9001 y RGPD.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Estructura de la tabla de auditoría):**
  **Dado que** el esquema de BD está definido,
  **Cuando** se revisa la tabla `audit_log`,
  **Entonces** contiene las columnas: `id` (UUID), `entity_type` (document/template/comment), `entity_id` (UUID), `block_uuid` (nullable), `action` (created/updated/deleted/state_changed/approved/rejected), `user_id`, `ip_address`, `user_agent`, `timestamp` (timestamp del servidor), `previous_value` (JSONB, nullable), `new_value` (JSONB, nullable).

- [ ] **Escenario 2 (Solo INSERT permitido al usuario de aplicación):**
  **Dado que** el usuario de aplicación Laravel intenta hacer UPDATE o DELETE en `audit_log`,
  **Cuando** ejecuta la sentencia,
  **Entonces** PostgreSQL devuelve un error de permisos insuficientes; el usuario de aplicación solo tiene permiso `INSERT` en esta tabla.

- [ ] **Escenario 3 (Timestamp siempre del servidor):**
  **Dado que** se registra cualquier acción en `audit_log`,
  **Cuando** se inserta la fila,
  **Entonces** el campo `timestamp` se establece con `NOW()` de PostgreSQL, nunca con un valor enviado por el cliente ni por el frontend.

- [ ] **Escenario 4 (Registro automático de cambios de estado):**
  **Dado que** un documento o plantilla cambia de estado,
  **Cuando** la transición ocurre,
  **Entonces** se inserta automáticamente un registro en `audit_log` con `action = 'state_changed'`, el estado anterior y el nuevo estado.

- [ ] **Escenario 5 (Consulta de historial con control de acceso):**
  **Dado que** un usuario solicita el historial de auditoría de un documento,
  **Cuando** el endpoint procesa la petición,
  **Entonces** solo el autor del documento, los revisores asignados o usuarios con rol de dirección reciben los datos; otros usuarios reciben HTTP 403.

- [ ] **Requisito Técnico (Inserción no bloqueante):**
  La inserción en `audit_log` debe completarse en menos de 10 ms para no añadir latencia perceptible al flujo principal.

**Notas:**
- La tabla de auditoría es relacional y separada del JSONB (Hecho 20, Decisión D-11). Esto permite queries SQL estándar sobre el historial.
- `AuditLogRepository::record(AuditLogEntry $entry)` es el único punto de escritura; todos los Services lo invocan al finalizar operaciones relevantes.
- Los backups diarios de `audit_log` se almacenan encriptados off-site para prevenir tampering por administradores de BD (STRIDE T-AUDIT-01).
