---

**Descripción:**
Como equipo de operaciones
Quiero endpoints de health check estandarizados que verifiquen el estado de todos los servicios dependientes de Maya PD
Para que el sistema de monitoreo externo detecte automáticamente degradaciones antes de que impacten a los usuarios.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (GET /health — estado de todos los servicios):**
  **Dado que** todos los servicios dependientes están operativos,
  **Cuando** el sistema de monitoreo consulta `GET /health`,
  **Entonces** el servidor devuelve HTTP 200 con body JSON que muestra estado `ok` para cada servicio: Laravel, PostgreSQL local, conexión FDW, Redis, RabbitMQ y WebSocket server.

- [ ] **Escenario 2 (GET /health con servicio degradado):**
  **Dado que** la conexión FDW no responde,
  **Cuando** el sistema de monitoreo consulta `GET /health`,
  **Entonces** el servidor devuelve HTTP 503 con el body JSON indicando `"fdw": { "status": "down", "latency_ms": null }` y los demás servicios con su estado real.

- [ ] **Escenario 3 (Liveness probe — GET /health/live):**
  **Dado que** el proceso Laravel está corriendo,
  **Cuando** el orquestador consulta `GET /health/live`,
  **Entonces** el servidor devuelve HTTP 200 con `{"status": "ok"}` sin verificar dependencias externas; solo confirma que el proceso está vivo.

- [ ] **Escenario 4 (Readiness probe — GET /health/ready):**
  **Dado que** el servidor está iniciando y las dependencias aún no están listas,
  **Cuando** el orquestador consulta `GET /health/ready`,
  **Entonces** devuelve HTTP 503 hasta que todas las dependencias críticas (BD local, Redis) estén disponibles; una vez listas, devuelve HTTP 200.

- [ ] **Escenario 5 (Endpoints sin autenticación JWT):**
  **Dado que** el load balancer necesita verificar el estado del servidor sin token,
  **Cuando** consulta `/health`, `/health/live` o `/health/ready`,
  **Entonces** los endpoints responden sin requerir cabecera `Authorization`; están excluidos del middleware de JWT.

- [ ] **Requisito Técnico (Tiempo de respuesta máximo):**
  El endpoint `/health` completo (incluyendo todos los checks con timeout individual de 500 ms) debe responder en menos de 2 segundos.

---

**MoSCoW:** `Should`
**Sprint:** Sprint 6

**Notas:**
- Usar `/health/live` y `/health/ready` separados para compatibilidad con Kubernetes probes y Docker healthchecks.
- El body de `/health` incluye la latencia de cada check individual para detectar degradaciones de rendimiento antes de fallo completo.
- `HealthCheckService::check()` ejecuta todos los checks en paralelo para reducir el tiempo total de respuesta.
- Los endpoints de health no están en el grupo de rutas API autenticadas; tienen su propio grupo de rutas públicas.
