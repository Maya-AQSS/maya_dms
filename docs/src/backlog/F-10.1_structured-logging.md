**Descripción:**
Como equipo de operaciones
Quiero que todos los componentes de Maya PD emitan logs estructurados en formato JSON con un Trace ID correlado entre la API y los Workers
Para poder diagnosticar incidencias en producción de forma rápida sin necesidad de acceso directo a la base de datos.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Formato JSON con campos estándar):**
  **Dado que** cualquier componente de Maya PD genera un log,
  **Cuando** el log se emite,
  **Entonces** tiene formato JSON con los campos: `timestamp` (ISO 8601), `level` (DEBUG/INFO/WARNING/ERROR), `trace_id`, `user_id` (del JWT, null si no autenticado), `action`, `entity_type`, `entity_id`, `message`.

- [ ] **Escenario 2 (Trace ID propagado de API a Worker):**
  **Dado que** la API encola un Job de generación PDF,
  **Cuando** el Worker procesa el Job,
  **Entonces** el log del Worker incluye el mismo `trace_id` que el log de la API que originó la petición, permitiendo correlacionar la petición con el resultado del Job.

- [ ] **Escenario 3 (Eventos de negocio críticos con nivel INFO):**
  **Dado que** ocurre un evento de negocio relevante,
  **Cuando** se registra el log,
  **Entonces** los siguientes eventos usan nivel INFO: autenticación JWT exitosa, publicación de documento, rechazo de validación, inicio y fin de generación PDF, y Job fallido tras 3 reintentos.

- [ ] **Escenario 4 (Datos sensibles nunca logueados):**
  **Dado que** el sistema procesa datos potencialmente sensibles,
  **Cuando** se generan logs de cualquier nivel,
  **Entonces** nunca se incluyen: tokens JWT completos, contraseñas, contenido JSONB completo de bloques, ni datos personales del FDW más allá del `user_id`.

- [ ] **Escenario 5 (Exportación a stdout para orquestadores):**
  **Dado que** la aplicación está desplegada en un entorno con Docker o Kubernetes,
  **Cuando** los logs se emiten,
  **Entonces** se escriben a stdout en lugar de a archivos de disco, permitiendo que el orquestador los capture y envíe a ELK/Loki.

- [ ] **Requisito Técnico (Overhead del logging):**
  El overhead de logging por evento debe ser inferior a 1 ms para no añadir latencia perceptible a los endpoints.

**MoSCoW:** `Should`
**Sprint:** Sprint 6

**Notas:**
- Los logs de auditoría de negocio (quién editó qué bloque) van en la tabla `audit_log` (F-09.1). Los logs de aplicación son para diagnóstico técnico operacional.
- Usar el canal de logging de Laravel configurado con el driver `monolog` y el formatter `JsonFormatter`.
- El `trace_id` se genera en el middleware de autenticación y se propaga en el Job payload para correlación entre API y Worker.
- El `user_id` en logs se refiere al `sub` del JWT (UUID), nunca al nombre o email del usuario (RGPD).
