**Descripción:**
Como colaborador de un documento
Quiero que el sistema bloquee automáticamente el bloque que estoy editando para los demás usuarios activos
Para evitar conflictos de edición simultánea y saber qué sección está ocupada por otro colaborador.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Solicitud y confirmación de lock):**
  **Dado que** un colaborador inicia la edición de un bloque,
  **Cuando** el cliente envía un mensaje de solicitud de lock al servidor vía WebSocket,
  **Entonces** el servidor valida el permiso del usuario, confirma el lock y notifica a todos los presentes en el canal del documento: el bloque aparece como "Bloqueado por [nombre]" con fondo diferenciado.

- [ ] **Escenario 2 (Intento de edición de bloque bloqueado):**
  **Dado que** un bloque está bloqueado por otro colaborador,
  **Cuando** un segundo usuario intenta editar ese bloque,
  **Entonces** el editor de ese bloque aparece deshabilitado con el mensaje "Bloqueado por [nombre del colaborador]" y no puede recibir input.

- [ ] **Escenario 3 (Liberación de lock por inactividad — timeout):**
  **Dado que** un colaborador tiene un lock activo pero lleva 5 minutos sin actividad de edición (sin heartbeat),
  **Cuando** el servidor detecta la expiración del timeout,
  **Entonces** el lock se libera automáticamente y el bloque vuelve a estar disponible; todos los presentes reciben la notificación de liberación.

- [ ] **Escenario 4 (Liberación de lock al salir):**
  **Dado que** un colaborador tiene locks activos en el documento,
  **Cuando** cierra la pestaña del navegador o navega fuera del editor,
  **Entonces** todos sus locks se liberan inmediatamente vía evento de desconexión del WebSocket.

- [ ] **Escenario 5 (Canal de presencia — usuarios activos visibles):**
  **Dado que** varios colaboradores tienen el documento abierto simultáneamente,
  **Cuando** cualquiera de ellos visualiza el editor,
  **Entonces** puede ver los avatares o nombres de todos los usuarios activos en el documento en ese momento.

- [ ] **Requisito Técnico (Latencia de notificación de lock):**
  La notificación de lock debe ser entregada a todos los presentes en el canal en menos de 100 ms.

**Notas:**
- El servidor (backend Laravel) es la fuente de verdad del lock; el cliente solo solicita y recibe confirmación; nunca gestiona el lock de forma autónoma.
- Si el WebSocket no está disponible, el sistema funciona sin locking colaborativo (degradación grácil); el editor sigue siendo funcional aunque sin protección de concurrencia.
- D-36: analizar en Sprint 2 si HTTP polling es más viable que WebSocket para el MVP; este backlog asume la implementación WebSocket.
- `DocumentLockService::acquireLock(string $documentId, string $blockUuid, string $userId)` gestiona el estado del lock en Redis con TTL.
- Heartbeat del cliente: ping al servidor cada 30 s para renovar el TTL del lock.
