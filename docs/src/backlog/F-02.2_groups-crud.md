**Descripción:**
Como jefe de departamento
Quiero gestionar grupos internos de usuarios en Maya PD (crear, editar, eliminar, asignar miembros)
Para organizar docentes y asignar visibilidad de plantillas a colectivos sin depender del sistema externo.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (CRUD completo de grupos):**
  **Dado que** un usuario con permisos de gestión está autenticado,
  **Cuando** utiliza los endpoints de grupos,
  **Entonces** puede crear (`POST /api/groups`), listar (`GET /api/groups`), editar (`PUT /api/groups/{id}`) y eliminar (`DELETE /api/groups/{id}`) grupos internos.

- [ ] **Escenario 2 (Relación N:N de miembros):**
  **Dado que** un grupo existe en el sistema,
  **Cuando** se asignan usuarios al grupo,
  **Entonces** un grupo puede tener múltiples usuarios y un usuario puede pertenecer a múltiples grupos simultáneamente (relación N:N sin anidamiento).

- [ ] **Escenario 3 (Eliminación no destructiva):**
  **Dado que** un grupo tiene usuarios asignados,
  **Cuando** se elimina el grupo,
  **Entonces** los usuarios no son eliminados del sistema; únicamente se desvincula la relación entre el grupo y sus miembros.

- [ ] **Escenario 4 (Control de acceso por rol):**
  **Dado que** un docente sin permisos de gestión intenta crear o eliminar un grupo,
  **Cuando** ejecuta la acción correspondiente,
  **Entonces** el sistema devuelve HTTP 403; cualquier usuario puede ver los grupos a los que pertenece pero solo los roles superiores pueden gestionarlos.

- [ ] **Requisito Técnico (Rendimiento de listado):**
  El endpoint de listado de grupos con sus miembros debe responder en menos de 200 ms para grupos con hasta 200 usuarios.

**Notas:**
- Los grupos son 100% internos de Maya PD y no se sincronizan con el sistema corporativo externo.
- Los grupos no tienen anidamiento: solo un nivel de profundidad (un grupo no puede contener a otro grupo).
