# SECURITY F-01.4 — IDOR Global Scopes

**Épica:** EPIC-01 — Autenticación, Sesión y Control de Acceso

## 🏷️ Categoría
**Categoría:** `Security`
**Impacta en:** Backend

## 📦 Dependencias
> Backlogs que deben estar **completados** antes de implementar esta feature.

| Backlog | Motivo | Bloqueante |
| --- | --- | --- |
| [F-01.1](F-01.1_jwt-validation.md) | Provee la identidad del usuario autenticado necesaria para aplicar los scopes de filtrado | Sí |

**Prioridad:** `Must`

---

**Descripción:**
Como arquitecto de seguridad
Quiero que todos los modelos Eloquent de Laravel apliquen Global Scopes que filtren recursos por los permisos del usuario autenticado
Para prevenir ataques de referencia directa a objetos (IDOR) y garantizar aislamiento de datos entre usuarios.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Global Scopes en modelos principales):**
  **Dado que** el sistema tiene modelos de datos principales,
  **Cuando** se revisa la definición de los modelos `Document`, `Template`, `Group` y `Comment`,
  **Entonces** cada uno tiene un Global Scope aplicado que filtra los registros para devolver únicamente aquellos a los que el usuario actual tiene acceso (como propietario, colaborador o recurso asignado).

- [ ] **Escenario 2 (Obligatoriedad de findOrFail):**
  **Dado que** un controlador necesita recuperar un recurso por su ID,
  **Cuando** se revisa el código de los controladores,
  **Entonces** todos usan `Document::findOrFail($id)` u equivalente que aplica el Global Scope, y ninguno usa `Document::find($id)` sin pasar por el scope de seguridad.

- [ ] **Escenario 3 (Respuesta 404 para recursos inaccesibles):**
  **Dado que** un usuario intenta acceder a un recurso que existe pero al que no tiene permisos,
  **Cuando** el Global Scope filtra ese recurso,
  **Entonces** el sistema devuelve HTTP 404 (no HTTP 403) para no revelar la existencia del recurso al atacante.

- [ ] **Escenario 4 (Test de aislamiento entre usuarios):**
  **Dado que** el usuario B intenta acceder a un documento del usuario A por su ID,
  **Cuando** ejecuta `GET /api/documents/{id_del_documento_de_A}`,
  **Entonces** el sistema devuelve HTTP 404 confirmando el aislamiento entre usuarios.

- [ ] **Requisito Técnico (Overhead del scope):**
  El Global Scope añade únicamente una cláusula WHERE a la consulta SQL. El overhead de rendimiento debe ser inferior a 0.5 ms por consulta.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 1

**Notas:**
- Usar HTTP 404 en lugar de HTTP 403 para recursos inaccesibles es una decisión deliberada de seguridad (principio de no revelar la existencia de objetos no autorizados).
- El Global Scope debe ser transparente: el resto del código de la aplicación no necesita añadir condiciones de seguridad manualmente.
