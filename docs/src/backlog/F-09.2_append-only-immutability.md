**Descripción:**
Como sistema de gestión documental
Quiero garantizar a nivel de código y de base de datos que las versiones publicadas de documentos y plantillas sean completamente inmutables
Para cumplir con el principio Append-Only requerido por ISO 9001 y prevenir alteraciones retrospectivas del historial.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Policy bloquea modificación de documento publicado):**
  **Dado que** un documento está en estado `published`,
  **Cuando** cualquier usuario intenta actualizar su contenido vía API,
  **Entonces** la `DocumentPolicy::update()` devuelve `false` y el servidor responde HTTP 403.

- [ ] **Escenario 2 (Constraint de BD en versiones inmutables):**
  **Dado que** un snapshot en `document_versions` tiene `is_immutable = true`,
  **Cuando** cualquier proceso intenta ejecutar UPDATE o DELETE sobre esa fila,
  **Entonces** PostgreSQL rechaza la operación mediante un trigger o check constraint; el `is_immutable` no puede cambiarse una vez establecido a `true`.

- [ ] **Escenario 3 (is_immutable se establece al publicar):**
  **Dado que** un snapshot es creado al publicar,
  **Cuando** el `SnapshotService` inserta el registro,
  **Entonces** el campo `is_immutable` se establece a `true` en el mismo INSERT y no existe endpoint ni método del Service que permita cambiarlo posteriormente.

- [ ] **Escenario 4 (Test de integración de inmutabilidad):**
  **Dado que** existe un documento publicado con snapshot,
  **Cuando** el test intenta modificarlo vía API con credenciales válidas de su autor,
  **Entonces** el test verifica que la respuesta es HTTP 403 y que el snapshot en BD no ha cambiado.

- [ ] **Escenario 5 (Versiones publicadas no se eliminan):**
  **Dado que** una nueva versión de un documento es publicada,
  **Cuando** el sistema gestiona el historial,
  **Entonces** las versiones anteriores permanecen en `document_versions` y son accesibles desde el historial; no se eliminan ni archivan automáticamente.

- [ ] **Requisito Técnico (Verificación de inmutabilidad sin overhead):**
  La verificación del campo `is_immutable` es una consulta de un campo booleano que debe completarse en menos de 1 ms.

**MoSCoW:** `Must`

**Notas:**
- La inmutabilidad se aplica a los snapshots (versiones publicadas), no al documento en sí. El documento puede publicar nuevas versiones; las anteriores son intocables.
- Doble protección: Policy de Laravel (capa de aplicación) + constraint de PostgreSQL (capa de datos). Ambas capas son necesarias para máxima seguridad.
- Restricción 1 y 2 del documento maestro son el fundamento normativo de este backlog.
