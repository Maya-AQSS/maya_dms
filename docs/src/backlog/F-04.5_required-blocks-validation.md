# [LOGIC / BUSINESS] F-04.5 — Validación de Bloques Obligatorios

**Épica:** EPIC-04 — Gestión de Documentos

## 🏷️ Categoría
**Categoría:** `Logic / Business`
**Secundaria:** `UI / Presentation`
**Impacta en:** Equipo Frontend / Backend

## 📦 Dependencias

| Backlog | Motivo | Bloqueante |

| [F-03.3](F-03.3_block-states-flags.md) | Provee el flag `obligatorio/opcional` por bloque que determina cuáles se validan | Sí |

**Prioridad:** `Must`

---

**Descripción:**
Como sistema
Quiero validar que todos los bloques marcados como obligatorios estén rellenos antes de permitir enviar el documento a revisión
Para evitar que programaciones incompletas entren en el flujo de aprobación y cumplir con los criterios de calidad de la plantilla normativa.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Botón deshabilitado con bloques vacíos):**
  **Dado que** un documento tiene al menos un bloque obligatorio vacío,
  **Cuando** el docente visualiza el editor,
  **Entonces** el botón "Enviar a revisión" aparece deshabilitado con un indicador que muestra el número de bloques obligatorios pendientes de rellenar.

- [ ] **Escenario 2 (Resaltado en el outline de bloques incompletos):**
  **Dado que** el docente intenta enviar a revisión con bloques obligatorios vacíos,
  **Cuando** pulsa el botón (si estuviera habilitado) o revisa el outline,
  **Entonces** los bloques obligatorios vacíos aparecen resaltados en el outline izquierdo con un indicador de error visual.

- [ ] **Escenario 3 (Validación backend independiente — HTTP 422):**
  **Dado que** una petición de envío a revisión llega al backend con bloques obligatorios vacíos (bypaseando la validación frontend),
  **Cuando** el `DocumentService` verifica el estado de los bloques,
  **Entonces** el servidor devuelve HTTP 422 con la lista de UUIDs de los bloques obligatorios vacíos.

- [ ] **Escenario 4 (Bloques Bloqueados siempre considerados rellenos):**
  **Dado que** un bloque tiene estado `Bloqueado` (contenido normativo de la plantilla),
  **Cuando** se evalúa si el documento puede enviarse a revisión,
  **Entonces** el sistema considera ese bloque como "relleno" independientemente de si el docente ha introducido contenido adicional.

- [ ] **Escenario 5 (Todos los obligatorios rellenos — envío habilitado):**
  **Dado que** todos los bloques obligatorios tienen contenido,
  **Cuando** el docente visualiza el editor,
  **Entonces** el botón "Enviar a revisión" está habilitado y el outline no muestra alertas de bloques incompletos.

- [ ] **Requisito Técnico (Velocidad de validación backend):**
  La validación de bloques obligatorios en el backend debe ejecutarse en menos de 50 ms.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 3

**Notas:**
- La validación dual (frontend + backend) es intencional: el frontend mejora la UX; el backend es la fuente de verdad.
- Un bloque se considera "relleno" si su contenido JSONB no es un array vacío ni contiene únicamente bloques de párrafo vacíos; el `DocumentService` define esta regla.
- `DocumentService::validateRequiredBlocks(string $documentId)` es invocado por `DocumentService::submitForReview()` antes de cambiar el estado.
