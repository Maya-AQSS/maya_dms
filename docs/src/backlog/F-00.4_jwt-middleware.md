**Descripción:**
Como sistema
Quiero un middleware Laravel que valide criptográficamente el token JWT emitido por el dashboard corporativo
Para garantizar que solo usuarios autenticados acceden a la API y que los claims de sesión son confiables bajo un modelo Zero Trust.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Protección de rutas):**
  **Dado que** la API de Maya PD está desplegada,
  **Cuando** se revisa la configuración de rutas,
  **Entonces** el middleware JWT está activo en todas las rutas de la API excepto en el endpoint `/health`.

- [ ] **Escenario 2 (Validación criptográfica):**
  **Dado que** el middleware recibe una petición con un token JWT,
  **Cuando** valida el token,
  **Entonces** verifica la firma RS256 contra el JWKS público del dashboard corporativo (cuya URL está en variable de entorno) y valida los claims `iss`, `aud` y `exp`.

- [ ] **Escenario 3 (Caché de JWKS):**
  **Dado que** el middleware necesita las claves públicas del dashboard corporativo,
  **Cuando** el dashboard corporativo no está disponible,
  **Entonces** el middleware utiliza las claves JWKS cacheadas en Redis con TTL de 1 hora para seguir validando tokens durante la indisponibilidad.

- [ ] **Escenario 4 (Token inválido o expirado):**
  **Dado que** un cliente envía una petición con un token JWT inválido o expirado,
  **Cuando** el middleware lo evalúa,
  **Entonces** la respuesta es un HTTP 401 con cuerpo JSON `{"error": "Unauthenticated"}` y no se procesa la petición.

- [ ] **Escenario 5 (Seguridad en logs):**
  **Dado que** ocurre un fallo de autenticación,
  **Cuando** el sistema registra el evento en los logs,
  **Entonces** se incluyen IP, User-Agent y los primeros 8 caracteres del token, pero nunca el token completo.

- [ ] **Requisito Técnico (Rendimiento):**
  El overhead del middleware debe ser inferior a 5 ms en el percentil 95 cuando el caché de JWKS está caliente en Redis.

**Notas:**
- No hay pantalla de login propia en Maya PD. El usuario llega ya con un token JWT emitido por el dashboard corporativo.
- El payload del JWT nunca se logea completo; solo `sub` y `exp` son campos aceptables en los registros de diagnóstico.
