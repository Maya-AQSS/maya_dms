**Descripción:**
Como sistema de gestión documental
Quiero versionar los documentos mediante snapshots inmutables en PostgreSQL siguiendo el principio Append-Only
Para conservar el historial completo de versiones publicadas y cumplir con la trazabilidad requerida por ISO 9001.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Creación de snapshot al publicar):**
  **Dado que** un documento pasa a estado Publicado,
  **Cuando** se confirma la publicación,
  **Entonces** se crea un registro en `document_versions` con el JSONB completo de todos los bloques en ese momento, el número de versión correlativo, el autor, el changelog y el timestamp del servidor.

- [ ] **Escenario 2 (UUID de bloque conservado entre versiones):**
  **Dado que** un documento tiene varias versiones publicadas,
  **Cuando** se comparan los bloques de distintas versiones,
  **Entonces** cada bloque mantiene su UUID original inmutable en todas las versiones; esto permite anclar comentarios y auditoría al UUID del bloque.

- [ ] **Escenario 3 (Inmutabilidad de versiones publicadas):**
  **Dado que** un snapshot en `document_versions` tiene `is_immutable = true`,
  **Cuando** cualquier actor intenta modificar ese registro,
  **Entonces** el sistema devuelve HTTP 403 (Policy) y la BD rechaza la operación (constraint de nivel de base de datos).

- [ ] **Escenario 4 (Historial de versiones consultable):**
  **Dado que** un documento tiene múltiples versiones publicadas,
  **Cuando** un usuario autorizado consulta el historial,
  **Entonces** recibe una lista cronológica con número de versión, fecha de publicación, autor y changelog de cada versión.

- [ ] **Escenario 5 (Documento activo referencia versión más reciente):**
  **Dado que** un documento tiene versiones v1.0 y v2.0 publicadas,
  **Cuando** un usuario abre el documento,
  **Entonces** ve el contenido de la versión más reciente (v2.0) por defecto y puede navegar al historial para ver versiones anteriores.

- [ ] **Requisito Técnico (Rendimiento de consulta por versión):**
  La query de consulta de una versión específica por `version_id` debe responder en menos de 100 ms gracias al índice sobre `(document_id, version_number)`.

**Sprint:** Sprint 3

**Notas:**
- El versionado es estrictamente Append-Only: nunca se modifica ni elimina un snapshot publicado (Restricción 1, ADR-01).
- La lógica de creación del snapshot reside en `SnapshotService::createDocumentSnapshot(CreateSnapshotDTO $dto)`.
- El `DocumentRepository` expone `findVersionById(string $versionId)` para consultas directas.
- El mensaje de changelog es obligatorio al publicar (HTTP 422 si vacío), heredado del comportamiento de plantillas (F-03.4).
